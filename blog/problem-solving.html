Problem Solving

  1. A story about a time I got blocked on a simple problem
    • There are countless times when I got blocked on a simple problem and the problem is usually a missing, extra or wrong character. For example a missing colon, or calling your variable testScore instead of testScores.

      The best way to solve these problems is to stay calm. Re-read through your code and hopefully spot your error

      However, it is sometimes hard to stay calm when you don't know it's just a simple error. What if there is something fundamentally wrong with your code?

      That's when the wider range of problem solving methods come in handy!
  2. A story about a time I solved a problem in an elegant way
    • The Zen Garden task got us to format a decent sized html file that was a mess.

      I was not a fan of this task at all. Going through it, line by line, trying to figure out how each line should be formatted was tedious and boring. If only I was a better programmer I could just automate this!

      Just as I was almost done, I stopped to have some lunch. After a nice lunch I came back to my computer and I had multiple VS code windows open for different tasks. To clean up my desktop, I closed the ones I wasn't working on, and also closed ZEN GARDEN WITHOUT SAVING.

      I was not ready to go through it all again. It was fine, I'll just skip that task, who cares... But I decided to give it another go but with a more tactical approach. I researched some handy keyboard shortcuts that would help me navigate the large document (namely jump to html tag pair and select content between html tag pairs) and re-did the tasks

      By formatting the document as a whole, instead of line-by-line and with my new keybindings in place. I managed to complete the task much more efficiently!
  3. Reflecting on how confident I feel using the problem solving techniques listed
    • Pseudocode
      • Relatively comfortable! It does help to layout some pseudocode before you write some code that is adventuring out of your comfort zone.

        I am unsure when to leave the pseudocode in the final code as comments, or remove them to clean up the file.
    • Trying something
      • I am getting more and more confident at this as I code more and have more finished code under my belt
    • Rubber ducky method
      • I should probably do this more often! When I do, do it, I do feel confident though.
    • Reading error messages
      • While a lot of error messages still seem very cryptic, I am getting better at reading them and taking away what is useful to me.
    • Console.logging
      • This is the problem solving technique I am most confident at and use it extensively. I still have room for improvement and could apply this problem solving technique in a more programatic way
    • Googling
      • I am also pretty confident at this however I get annoyed at the large variety of websites you end up visiting that all have different layouts and functionality. Finding one website to replace them all (mostly) would be awesome!
    • Asking your peers for help
      • I am not confident in this yet as I have not run into any problems that have completely stumped me yet. I look forward to the day though!
    • Asking coaches for help
      • See above.
    • Improving your process with reflection
      • Improving in this with each problem I face! It really does help alot.
